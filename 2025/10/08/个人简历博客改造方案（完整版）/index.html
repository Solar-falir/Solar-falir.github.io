🎯 个人简历博客改造方案（完整版）

一、整体定位

将 Researcher 学术主题改造为**”专业技术人员个人简历展示平台”**，突出：

  • 职业履历的完整性和专业性
  • 技术能力的可视化展示
  • 认证资质的权威背书
  • 联系方式的多元化和便捷性

核心理念：最小化代码修改，充分利用现有主题设计，通过数据层+少量布局层调整实现转型。

二、仓库分析

主目录结构分析

路径 类型 功能说明
_config.yml 文件 [Hexo核心] 站点主配置文件。包含网站标题、作者、URL、主题选择等全局设置。
_config.hexo-theme-researcher.yml 文件 [主题] Researcher 主题的独立配置文件，用于覆盖主题的默认设置，实现配置分离。
package.json 文件 [Node.js] 定义项目依赖（如Hexo版本）和脚本。
source/ 目录 [Hexo核心] 存放所有原始内容。_posts放文章，_data放数据文件，其他子目录对应独立页面。
themes/ 目录 [Hexo核心] 存放所有主题。researcher 主题就安装在这里。
scaffolds/ 目录 [Hexo核心] 模板文件目录。hexo new 命令会基于此目录中的文件创建新文章或页面。
.deploy_git/ 目录 [Hexo部署] hexo deploy 命令生成的目录，用于存放即将推送到GitHub Pages的静态网站文件。
public/ 目录 [Hexo生成] hexo generate 命令生成的静态文件目录，包含了完整的网站内容。通常不纳入版本控制。
.gitignore 文件 [Git] 定义了需要被Git忽略的文件和目录，如 node_modules 和 public。
temp_data/ 目录 [用户自定义] 用户存放的临时数据，目前已被.gitignore忽略。

主题目录结构分析 (themes/researcher)

路径 类型 功能说明
_config.yml 文件 [主题核心] 主题的默认配置文件。我们通过根目录的 _config.hexo-theme-researcher.yml 来覆盖它。
package.json 文件 [主题依赖] 定义了主题自身的依赖（如Tailwind CSS, DaisyUI）和构建脚本（如build:css）。
tailwind.config.js 文件 [主题样式] Tailwind CSS 的配置文件，定义了主题的颜色、字体、间距等设计系统。样式定制的核心。
layout/ 目录 [主题核心] EJS 模板文件目录，定义了网站的HTML结构。layout.ejs是主布局，其他*.ejs对应不同页面。
source/ 目录 [主题资源] 存放主题的前端资源，如CSS、JavaScript、图片等。source/css/tailwind.css是样式源文件。
demo_data/ 目录 [主题示例] 提供了.yml数据文件的示例，方便用户理解数据结构。

三、主题深度解析

核心技术栈

技术 角色 功能说明
EJS 模板引擎 动态渲染。将 source/_data 中的数据和 source/_posts 中的文章内容，嵌入到 HTML 结构中，生成最终的静态网页。
Tailwind CSS CSS 框架 样式构建。提供原子化的 CSS 类，允许在 EJS 文件中直接组合这些类来快速构建界面，实现灵活的样式定制。
DaisyUI UI 组件库 视觉风格。作为 Tailwind CSS 的插件，它提供了预设的、美观的 UI 组件（如卡片、按钮），构成了主题的整体外观。
npm Scripts 构建脚本 样式编译。通过 package.json 中的 build:css 命令，将 Tailwind 配置和 EJS 中的类编译成最终的 style.css 文件。

关键文件/目录功能详解

  • layout/ (布局目录)

    • layout.ejs: 母版页。这是网站所有页面的“骨架”，定义了通用的头部(<head>)、页头(Header)、页脚(Footer)和侧边栏(Sidebar)结构。其他所有页面内容都会被嵌入到这个文件的 <%- body %> 标签中。
    • index.ejs, cv.ejs, projects.ejs 等: 页面模板。分别对应首页、CV页、项目页等。它们定义了各个页面的独特内容结构，并负责读取和循环 source/_data 中对应的 .yml 数据文件，然后将数据渲染成列表或卡片。
    • partials/: 可复用组件。存放被多个页面引用的“局部”模板，如 header.ejs (定义<head>内容), sidebar.ejs (侧边栏), footer.ejs (页脚)。修改这些文件会影响到所有引用它们的页面。
    • post.ejs: 文章详情页模板。用于渲染 source/_posts 目录下的单篇博客文章。

  • source/ (资源目录)

    • css/tailwind.css: 样式源文件。Tailwind CSS 在这里定义了基础指令和层。build:css 脚本会读取这个文件，并根据 tailwind.config.js 和 .ejs 文件中的类，生成最终的 style.css。
    • css/style.css: 最终样式文件。由构建脚本生成的、包含了所有网站样式的 CSS 文件，最终被 layout.ejs 引用。
    • js/: 存放主题所需的 JavaScript 文件，如 screenshot.js。
    • images/: 存放主题自带的图片，如默认头像、咖啡图标等。

  • _config.yml (主题默认配置文件)

    • 功能: 定义了主题的所有可配置项，如菜单链接、社交账号、广告位、DaisyUI 皮肤选择等。它的所有设置都可以被根目录的 _config.hexo-theme-researcher.yml 文件中的同名设置所覆盖。

  • tailwind.config.js (样式配置文件)

    • 功能: 样式定制的核心。您可以在这里扩展或修改主题的颜色、字体、边距、断点等视觉设计。例如，您可以添加一个新的品牌色，然后在 EJS 文件中通过 bg-brand-new-color 来使用它。修改后需要运行 npm run build:css 来使其生效。

四、页面功能与文件定位分析

1. CV (个人简历) 页面

  • 页面功能: 用于展示一份结构化的、全面的个人简历。
  • 改造用途: 完美契合，直接用于展示您的教育背景、工作经历、技能栈、所获奖项和语言能力。
  • 数据文件位置: source/_data/cv.yml
  • 布局文件位置: themes/researcher/layout/cv.ejs
  • 核心数据字段: education, experience, skills, awards, languages。

2. Projects (项目经历) 页面

  • 页面功能: 以卡片形式展示一系列项目。
  • 改造用途: 完美契合，作为核心的作品集展示区，详细呈现您参与过的重要项目。
  • 数据文件位置: source/_data/projects.yml
  • 布局文件位置: themes/researcher/layout/projects.ejs
  • 核心数据字段: title, status, period, category, description, tags, collaborators, featured。

3. Publications (技术分享) 页面

  • 页面功能: 设计为展示学术出版物列表。
  • 改造用途: 可灵活改造为您的技术文章列表。将每一篇技术博客或开源项目文档视为一次“发表”。
  • 数据文件位置: source/_data/publications.yml
  • 布局文件位置: themes/researcher/layout/publications.ejs
  • 核心数据字段: title, authors, journal (可改为“发表平台”), year, pdf (可改为文章链接), code (代码仓库链接)。

4. Talks (认证培训) 页面

  • 页面功能: 设计为展示学术演讲或报告。
  • 改造用途: 可改造为**“认证与培训”**展示墙，将每一次认证或重要培训视为一次“Talk”。
  • 数据文件位置: source/_data/talks.yml
  • 布局文件位置: themes/researcher/layout/talks.ejs
  • 核心数据字段: title, event (可改为“认证机构”), date, location, slides (可改为证书链接)。

5. Contact (联系方式) 页面

  • 页面功能: 提供一个联系信息展示页。
  • 改造用途: 直接使用，并可通过修改布局文件进行扩展，以增加微信、掘金等国内社交平台的联系方式。
  • 数据文件位置: 联系信息主要在主题配置文件 _config.hexo-theme-researcher.yml 的 contact 和 academic_profiles 部分定义。
  • 布局文件位置: themes/researcher/layout/contact.ejs

6. Notes (博客分享) 页面

  • 页面功能: 一个标准的博客文章列表页。
  • 改造用途: 作为您的个人博客或随笔区域，用于分享技术之外的思考、学习笔记或生活感悟。
  • 数据文件位置: 本页面不使用 _data 中的数据文件。它会自动抓取 source/_posts/ 目录下的所有 Markdown 文章。
  • 布局文件位置: themes/researcher/layout/notes.ejs

五、专家建议与最佳实践

1. 高效的内容工作流

  • 利用本地服务器实时预览: 在对网站进行任何修改（无论是修改配置、填充数据还是撰写文章）时，都应先在本地运行 hexo server --debug 命令。这会启动一个实时刷新的本地网站，您在编辑器中的任何保存操作，都会立刻在浏览器中看到效果，极大地提升了调试和修改的效率。
  • 善用草稿功能: 对于尚未完成或不希望立即发表的技术文章，应使用 hexo new draft "文章标题" 来创建。草稿文章存放在 source/_drafts 目录下，不会被 hexo generate 命令生成，确保了线上内容的整洁。当文章完成后，使用 hexo publish post "文章标题" 即可将其正式发布到 source/_posts 目录。
  • 区分数据与内容: 明确 source/_data 和 source/_posts 的用途。_data 中的 .yml 文件用于存放结构化、重复性的信息（如一条条的项目经历、一份份的证书），而 posts 中的 .md 文件用于存放非结构化、长篇幅的内容（如一篇完整的技术博客）。

2. 优雅的资源管理 (图片/文件)

  • 全局资源: 对于全站公用的图片（如个人头像），建议统一存放在根目录的 source/images 文件夹下，通过 /images/your-image.jpg 的方式引用。
  • 文章专属资源: 对于某篇技术文章内专属的插图，Hexo 提供了“文章资源文件夹”功能。将根目录 _config.yml 中的 post_asset_folder 设置为 true 后，每当您用 hexo new post "文章标题" 创建新文章时，会同时创建一个同名的文件夹。您只需将图片放入该文件夹，并在文章中直接用 ![图片描述](图片名.jpg) 的方式引用即可，无需关心路径问题。
  • 可下载文件: 对于简历PDF等需要提供下载的文件，建议在根目录创建 source/files 文件夹，将文件放入其中，并通过 /files/your-cv.pdf 的方式提供链接。

3. 理解双配置文件结构

在开始修改前，必须理解Hexo的“双配置”系统，这是保证项目可维护性的关键。

  • _config.yml (站点总司令部): 这是整个网站的主配置文件，负责站点的全局信息，如网站标题、作者，以及最重要的——指定使用哪个主题（theme: researcher）。

  • _config.hexo-theme-researcher.yml (主题装修方案): 这是主题专属的配置文件，负责主题内部的细节，如导航菜单的构成、社交链接、配色方案等。

  • 核心关系：“分离”与“覆盖”: Hexo优先使用根目录的 _config.hexo-theme-researcher.yml 来覆盖主题文件夹内部的默认配置。这使得您的个性化配置与主题文件本身相分离。当主题更新时，您可以直接替换主题文件夹，而您的所有定制化配置都会被完整保留并应用到新主题上，极大地便利了长期维护。

4. 注意事项：数据与模板的严格匹配

在本次调试中，我们发现了一个核心问题：即使数据存在，首页的“推荐”板块也无法显示。

根本原因在于，数据文件（如 projects.yml）中的 featured 字段，我们之前可能习惯性地设为布尔值 true。然而，本主题的模板文件 (index.ejs) 使用了严格相等 (===) 的判断条件，它期望的是字符串 'enable'。

结论与规范：未来所有需要“推荐”功能的数据条目，其 featured 字段必须设置为字符串 'enable'，而不是布尔值 true。

5. 注意事项：首页板块的显示逻辑

在调试中我们还发现，首页的 Research Interests 板块（我们用作“技术领域总结”）也一度无法显示。

根本原因在于，通过分析主题的 index.ejs 模板，我们发现其显示条件是检查 source/_data/cv.yml 文件中是否存在一个名为 research_interests 的顶级字段。

结论与规范：本主题的部分首页板块，其显示与否，是由特定数据字段是否存在来隐式控制的。如果未来需要隐藏该板块，只需在 cv.yml 文件中将 research_interests 字段整个注释掉或删除即可；反之，则需要确保该字段存在。

十一、内容更新与维护指南

1. 内容增删改基本原则

本站点的核心内容，除博客文章外，均由 source/_data/ 目录下的 .yml 文件驱动。这些文件是结构化的纯文本，请遵循 YAML 语法进行修改。

  • 新增条目: 每个条目都是一个以 - 开头的块。新增时，请完整复制一个“模块框架”，并粘贴到文件的适当位置（通常是顶部或底部）。
  • 修改条目: 直接找到对应条目，修改冒号 : 后面的值即可。
  • 删除条目: 找到要删除的条目，将其从 - 开始的整个块全部删除。
  • 注意事项: 修改时请注意缩进，YAML 语言对缩进非常敏感。建议使用专业的代码编辑器（如 VS Code）进行修改。
